# Frontend follow-up — sharing, scale, and the polish `docs/archive/ui-plan.md` left open

The admin island, self-hosted display font, and real alt text all shipped (see `docs/archive/ui-plan.md`). This doc covers what's next: the site currently has no way to be *shared* as individual headlines (no permalinks, no social preview cards, no feed syndication), which matters a lot for a parody headline site — that's the whole distribution mechanism for something in the FAX Sports/Onion mold. It also covers what's left of the original visual/a11y punch list, and the handles site's UI, which nothing has touched yet.

## 1. Sharing & discovery — the biggest gap

Right now `web/` has no `public/` directory at all — no favicon, no `robots.txt`, no sitemap, no Open Graph tags, and `/` is the only page (no per-headline permalink). Concretely:

- **Per-headline permalink page** (`web/src/pages/h/[id].astro` or similar): a single published headline, its stat block, photo, and source-note, on its own URL — what actually gets shared/screenshotted, rather than a link to the whole feed. Query via a new `getHeadlineById` in `web/src/lib/db.ts` (guard on `status = 'published'` — a draft's URL must 404, not leak an unreviewed headline).
- **Open Graph / Twitter Card meta tags** on that permalink page (`og:title`, `og:image`, `og:description` from `stat_block`) — this is what makes a shared link render as a card on Bluesky/Twitter/iMessage instead of a bare URL. `web/src/layouts/Base.astro` currently sets none of this; extend its `Props` to accept optional OG fields per-page rather than hardcoding site-wide values.
- **The parody label on the OG image itself is worth a second look against `SPEC.md`'s labeling philosophy.** The spec explicitly rules out "a card-graphic watermark" as content-level friction — but an OG card is arguably site-level (it's metadata about the page, not baked into the shareable headline text itself, and doesn't survive a screenshot of the headline card the way a watermark would). This is a judgment call, not an obvious yes/no — flag it for a decision rather than assuming either way when this gets built. Recommendation below leans toward including it, for what it's worth.
- **`robots.txt` + `sitemap.xml`**: trivial, and `/admin` specifically should be disallowed in `robots.txt` as one more (non-security) layer — Cloudflare Access already blocks crawlers from *reading* it, but there's no reason to even advertise the path.
- **RSS/Atom feed** of published headlines (`/feed.xml`): cheap to add given `getPublishedHeadlines` already exists, and fits a headline site better than most — lets people subscribe without needing an account (which is explicitly out of scope per `SPEC.md`).
- **Favicon**: currently none; `Base.astro`'s `<head>` has no `<link rel="icon">` at all, so browsers show a broken/default icon.

### Dynamically-generated OG images — a real per-headline card, not just a static photo

A static `og:image` (just the stored `photo_ref`) works, but it means every shared headline previews as a bare photo with no headline text on it — the thing that actually gets read at a glance in a Bluesky/Discord/iMessage unfurl. The stronger version composites the headline text + stat block + photo into one purpose-built image per headline, the way a real news-parody card would look. `karanshukla/navyfragen-app`'s `opengraph-service` + `server/src/lib/image-generator.ts` is a working reference for exactly this pattern (dynamic per-entity OG images, plus a related user-facing "export as image" feature) — worth reading directly before building this, not just this summary:

**The pattern, in general:** build the card as an HTML+CSS string server-side (a `<div>` with the photo as background, headline as overlaid text, stat block, footer wordmark), POST it to a headless-render service that screenshots it to PNG at a fixed size (`1200×630`, the standard OG dimensions — see `opengraph-service/internal/shim/shim.go`'s `OGWidth`/`OGHeight` and `BuildOGTemplate`), then serve that PNG as the `og:image`. The alternative (compositing at the pixel/canvas level with `sharp`/`canvas`) is more fiddly for text layout — rendering real HTML/CSS and screenshotting it is the easier way to get headline text to wrap and scale the way it does in the live card UI, since the same CSS can be reused.

**What navyfragen adds on top of the basic pattern, and which parts actually apply here:**
- *Crawler-gated generation* (`Classify` in `shim.go`): only render when the request's User-Agent matches a known link-preview bot (Bluesky's `Bluesky Cardyb`, plus Discord/Slack/Twitter's respective UAs) hitting a specific content path — everything else passes through unchanged. **Directly applicable** — a permalink page's normal HTML response doesn't need to carry a freshly-rendered image; only the crawler fetching the page for its unfurl does.
- *A whole separate Go proxy service sitting in front of the app, embedding its own Caddy instance* — this is solving a problem bluejays.space doesn't have: navyfragen resolves *arbitrary external* AT-Protocol profiles (any DID, not rows it controls), so it needs a language-agnostic shim that can front literally any client framework and a cache keyed by an external, attacker-influenceable identifier (handle → DID resolution, hit by a UA anyone can spoof). bluejays.space's content is a fixed, admin-curated set of rows already in its own Postgres — **the crawler-UA check and the render/cache step can both just live inside `web`'s existing Astro middleware and permalink route**, no second service, no embedded reverse proxy.
- *Disk-cached PNGs keyed by DID, TTL + LRU eviction, singleflight-deduped renders* — the caching *concept* is directly applicable (don't regenerate the same card on every crawler hit), but the storage and invalidation can be simpler here: cache in the **existing MinIO bucket** (already wired up for photos, already has a `web` proxy route) rather than a new disk volume, and key the cache entry on a hash of the fields that feed the template (headline text, stat block, photo_ref) rather than a bare TTL — since an edit to a published headline is a known, in-band event (an admin save), not an external system's independent mutation the way a Bluesky profile update is. A content hash sidesteps needing a new `updated_at` column on `headlines` (there isn't one today — only `created_at`/`published_at`) purely for cache invalidation; add one later only if something else ends up needing it too. This makes invalidation exact instead of "stale for up to N days." Singleflight-style de-dup (one render in flight per headline, concurrent requests for the same one share it) is still worth keeping regardless — cheap insurance against a viral headline getting hit by several crawlers at once before the first render finishes.
- *A concurrency cap on the render path* (`MaxConcurrentGenerate` in `handler.go`) guarding against a spoofed-UA render-storm exhausting a single shared render backend — still worth carrying over here if the render step calls out to any external headless-render dependency (see below), for the same reason: `Classify`'s UA check is a hint, not an auth boundary, and a hostile client can send any UA string it wants.

**What still needs a decision, not just a port:** the render step itself needs *something* that turns HTML into a PNG. navyfragen calls a sibling `html-to-image` microservice (a fourth Railway service, POST `{source, format, options}` → PNG bytes) — reasonable at their scale, since they already run several services and the render backend is shared by two features (OG cards and user-facing shareable images). For bluejays.space, standing up a whole extra headless-browser service for one feature may not be worth it; a Satori-based approach (`@vercel/og`/`satori` + `resvg`, or Astro's own `astro-og-canvas` integration) renders JSX/HTML-like input straight to SVG→PNG **in-process**, no headless browser, no extra Railway service, no `EXPORT_HTML_URL` network hop — likely the better starting point given `web` is a single Astro/Node service today. Revisit the separate-render-service approach only if Satori's layout engine turns out to be too limited for the actual card design (it doesn't support full CSS — flexbox/grid subset only, per Satori's own docs, worth checking against the real card layout before committing either way).

## 2. Public feed pagination

`getPublishedHeadlines()` has no `LIMIT`/offset and `index.astro` renders every published row in one page load. Fine today; not fine once the feed has months of headlines. Two reasonable shapes:
- Classic `?page=N` with a `LIMIT`/`OFFSET` query and prev/next links — simplest, works with zero JS, fits the "public pages ship zero/minimal JS" mandate in `SPEC.md`.
- Infinite scroll — better UX for a scrollable card feed, but needs a client-side island (breaking the zero-JS-on-public-pages posture) and a "load more" API route. Given the spec's explicit preference for minimal JS on public pages, **prefer classic pagination** unless there's a specific reason to reconsider.

Pick a page size (25-50 cards is reasonable for a card-per-headline feed) and land it alongside `docs/backend-api-plan.md` item 7 — the query and the page need to change together.

## 3. Handles site UI

`handles/templates/index.html` + `handles/static/` (Go service, server-rendered HTML + a small JS spinner for async PR-creation polling) hasn't had any design pass — it predates the Fraunces/Tailwind work done on `web`. Since it's a separate Go binary with its own template, it can't just inherit `web`'s Tailwind build; either hand-write matching CSS or accept it'll look visually distinct from the main site (it already lives on a different subdomain, `handles.bluejays.space`, so some visual distinction is less jarring than it would be on the same domain). Low priority relative to items 1-2, but worth a pass before treating the handles feature as "done" rather than "functional."

## 4. Remaining items from the archived UI plan, not yet re-verified

`docs/archive/ui-plan.md` flagged three things as still-open when the island/font/alt-text work landed; none of them were touched since, so they carry forward as-is:
- The Pudding-style visual/spacing pass, done against real photographed content (not stub text) — hasn't happened, since ingest was still stub-only when that doc was written and real drafts are only just starting to flow now that generation is real.
- A full WCAG contrast audit specifically on the register/fact-anchored badges (`bg-amber-100`/`text-amber-800`, `bg-red-100`/`text-red-800` in `DraftCard.svelte`) — never actually run against a contrast checker, just asserted as "worth checking."
- A deliberate mobile walkthrough with real (non-stub) content, since register-2 copy and real photos stress-test wrapping/truncation differently than placeholder text.

## 5. Dark mode / theming

Not in `SPEC.md` at all — not recommending it, just noting it's untouched (`global.css` has no `prefers-color-scheme` handling) in case it comes up as a request later. Don't build speculatively.

## Out of scope here

- Native mobile app, user accounts — out of scope per `SPEC.md` → Out of Scope (v1), unchanged.
